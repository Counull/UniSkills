# 需求关键词上下文补全

## 目标
当用户只给出一个需求关键词或功能名时，先从个人目录回溯到对应的需求分析与原始需求文档，补齐实现或评审所需上下文，再继续后续编码、分析或文档工作。

默认个人目录根路径:
- `D:/Project/MC/doc/6_个人目录/刘鸿森`

默认检索入口:
- `02_功能索引/功能索引总览.md`
- `01_需求分析/`
- `03_方案设计/`
- `00_收件箱/`

## 适用场景
- 用户只说“看一下政令需求”“继续做某个功能”，但没有贴完整需求背景。
- 需要从个人沉淀文档里先恢复需求语义，再开展实现、review 或方案分析。
- 需要判断一个功能已经有哪些分析结论、代码入口、风险与待确认项。
- 需要沿着“功能索引 -> 需求分析 -> 原始需求/知识库文档”追回来源依据。

## 输入参数
- 关键词或功能名（必填）
- 当前目标（可选）: 补上下文 / 开始实现 / 代码评审 / 更新分析文档
- 版本信息（可选）: 如 `V1.2`
- 个人目录根路径（可选）: 默认使用上述固定路径

## 执行流程
1. 规范化输入
- 提取主关键词、别名、版本号、目标动作。
- 若用户只给一句自然语言，先把其中最可能的功能关键词显式写出来再检索。

2. 优先查功能索引
- 先读 `02_功能索引/功能索引总览.md`。
- 优先匹配“功能名称”列，其次匹配“需求文档/设计文档/代码入口”列中的关键词。
- 命中后提取该行的:
  - 功能名称
  - 需求文档链接
  - 设计或分析文档链接
  - 代码入口
  - 状态与更新时间

3. 回读需求分析文档
- 若索引行存在分析文档，优先读取分析文档，因为它通常已经把原始需求拆成了研发可执行结构。
- 重点提取:
  - 背景与目标
  - 范围（In Scope / Out of Scope）
  - 核心规则、阈值、优先级
  - 数据/配置影响
  - 代码影响模块
  - 风险、待确认项、验收点

4. 回溯原始需求来源
- 在分析文档中优先寻找 `需求来源`、知识库引用、PRD/Docx 链接。
- 如果分析文档引用了原始需求文档，则继续读取该来源；若来源不可直接读取，也至少记录其路径和未验证状态。
- 如果需求来源是 docx、外部知识库导出或其他非纯文本格式:
  - 优先找是否已有 markdown 分析稿、引用摘录或同目录旁路文档可读。
  - 必要时再使用工具提取正文，不要臆造源文细节。

5. 组装“可执行上下文”
- 输出时至少整理出以下块:
  - 需求原意与业务目标
  - 不可违背的规则/数值/文案 key
  - 现有分析结论
  - 当前代码入口与潜在影响模块
  - 仍未确认的问题
- 如果用户后续要你实现、review 或写文档，就把这些内容直接作为后续工作的起始上下文，不再重复广泛探索。

6. 必要时衔接产文技能
- 如果已经找到原始需求来源，但个人目录中还没有对应分析文档，转而使用 `req-analysis-to-personal-doc` 继续生成正式分析稿。
- 如果已有分析文档，则优先复用已有结论，只补缺口，不重复造文档。

## 分支逻辑
- 命中多个候选功能:
  - 优先级: 功能名称精确匹配 > 功能索引命中 > 最近更新时间更新 > 未归档文档。
  - 默认选择最优候选，同时把其余候选简要列出，避免错绑功能。

- 功能索引未命中，但需求分析目录命中:
  - 直接读取最接近关键词且日期最新的分析文档。
  - 明确说明“未在功能索引中命中，本次基于需求分析目录推断”。

- 只有分析文档，没有原始需求来源:
  - 使用分析文档作为当前可信上下文。
  - 标记“原始需求未回溯验证”。

- 原始文档存在但不可读:
  - 先基于分析稿继续工作。
  - 仅把原始文档作为待验证来源，不擅自补全细节。

- 完全未命中:
  - 回退搜索 `03_方案设计` 与 `00_收件箱`。
  - 仍无结果时，要求用户补一个别名、版本号或更接近需求文档标题的关键词。

## 质量标准
- 先用个人目录里的现成结构恢复上下文，不一上来全仓乱搜。
- 明确区分“已由原始需求验证”“仅来自分析稿”“当前仍待确认”。
- 输出必须能支撑下一步动作，而不是只列文件名。
- 如果分析文档已经给出代码入口，后续代码检索从这些入口开始。
- 如果发现分析稿与代码现状冲突，明确指出冲突点，不默默覆盖原结论。

## 完成标准
- 至少定位到一个可信候选功能。
- 给出该功能的需求来源、分析文档、代码入口或缺失情况。
- 输出一份可直接用于后续实现/评审的上下文摘要。
- 若上下文不足，明确下一条最小补充信息，而不是泛泛要求“提供更多信息”。

## 推荐输出结构
1. 命中的功能与依据
2. 原始需求与分析结论
3. 代码入口与影响范围
4. 当前待确认项
5. 接下来直接执行的动作

## 示例触发
- `/mc-requirement-context 关键词=政令, 目标=补齐需求上下文`
- `先按政令这个词，把个人目录里对应需求和原始文档上下文补齐`
- `从我的功能索引里找政令需求，带上初始文档结论再开始改代码`

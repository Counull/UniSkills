# Code And Document Roast

这个 Skill 用来做一件事：

把“代码里真实发生的事”和“作者嘴上说的事”放在一起比，然后不留情面地指出哪些地方在自我美化、偷换概念、拿局部事实冒充全局判断。

它不是普通 code review。
它的重点不是格式、命名、语法糖。
它的重点是职责、边界、事务、生命周期、状态流、同步协议，以及文档是否在替坏结构辩护。

## When To Use

在这些场景下使用：

1. 用户要求“锐评”某个类、模块、系统或流程。
2. 用户要求“结合代码批判文档、方案、复盘、架构说明、Notion 页面”。
3. 用户怀疑现有代码在靠约定、顺序或局部补丁维持，而不是靠清晰边界维持。
4. 用户想拆穿某篇文档里哪些地方在美化旧设计。

不要在这些场景下使用：

1. 只是普通修 bug。
2. 用户只想要温和总结。
3. 用户只要命名/格式建议。
4. 没有明确目标代码或文档，且也没有可落脚的实现面。

## Inputs

输入通常有两类：

1. 目标代码：一个类、模块、流程入口、状态机节点、提交路径、UI 交互类。
2. 目标文档：Notion 页面、复盘文档、架构文档、设计说明、经验总结。

如果同时给了文档和代码，优先做“文档主张 vs 代码现实”的对照批判。

## Procedure

### A. 锐评代码

1. 先读目标文件。
2. 只扩散到最关键的 3 到 5 个邻接文件：
   - 调用方
   - 写回方
   - UI 依赖方
   - 状态切换方
   - 手动同步方
3. 找出最重的 3 到 6 个问题，优先级如下：
   - 职责混装
   - 事务边界不清
   - 生命周期靠隐式约定维持
   - 同一语义出现多套机制
   - 派生值被缓存成状态
   - UI、状态机、数据提交彼此缠绕
   - 手动同步点过多
   - 局部修补掩盖设计债
4. 先给一句定性狠话，再列证据。

### B. 结合代码锐评文档

1. 先读文档全文。
2. 抽出文档里的关键主张，特别关注这些句型：
   - “我们采用了……”
   - “这样可以……”
   - “逻辑和视图解耦……”
   - “易于扩展/维护/序列化/复用……”
   - “主要问题在于……”
3. 回仓库找对应代码证据，只围绕这些主张检验现实，不做无边界游览。
4. 优先指出：
   - 文档在美化什么
   - 文档漏掉了什么真正的问题
   - 文档把什么局部事实说成了全局事实
   - 文档把什么阶段性妥协包装成了设计判断
5. 如果代码现实明显更差，要直接说“这段文档已经不是过时，而是在替坏结构辩护”。

## Decision Points

### 什么时候继续扩散读文件

只有在下面几种情况下才继续多读：

1. 还无法确认行为到底由谁决定。
2. 还无法判断状态是在哪里写回的。
3. 还无法证明 UI 是否只是展示，还是已经参与业务门禁。

一旦已经能证明主张和现实不一致，就停止扩散，开始输出。

### 什么时候提高批判强度

看到这些模式时，默认提高批判强度：

1. 一个对象同时承担输入、预测、提交、回调、缓存、UI 前置判断。
2. 存在“先弹窗，再提交数据，再切状态”的链路。
3. 容器变化、关系变化、集合变化需要手动同步。
4. 文档宣称“解耦”，代码却依赖 ScriptBus、Dialog、状态机顺序。
5. 系统自称“数据驱动”，关键行为却依赖 UI 交互时机。
6. 缓存的是派生值，不是事实快照。
7. 事件、属性通知、局部 hook、状态切换多套语义并存。

### 什么时候收敛建议

如果用户要的不只是批判，还要下一步方向：

1. 只给 2 到 3 条最硬的建议。
2. 优先建议“停掉哪类自我欺骗式表述”。
3. 优先建议“压缩哪类职责”。
4. 优先建议“消灭哪类隐式协议”。
5. 不要给大而化之的重构口号。

## Output Format

按下面顺序输出，除非用户明确要求别的格式。

### 一句话结论

先用一句狠话定性，但不能空洞。

示例风格：

1. 这不是轻微耦合，而是把事务、UI 和状态切换揉成了一个靠顺序运气维持的流程对象。
2. 这篇文档不是在复盘，而是在给旧设计留体面。

### 主要问题

列 3 到 6 条最重的问题。每条都要包含：

1. 问题定性。
2. 为什么严重。
3. 代码证据。

### 代码证据

只列最关键的文件和调用关系，优先：

1. 决定行为的地方。
2. 写回状态的地方。
3. UI 或状态机介入业务的地方。
4. 手动同步或局部补丁的地方。

### 文档批判

仅在用户给了文档时输出，使用这个句式：

1. 原文主张。
2. 现实代码。
3. 更准确的说法。

### 收口建议

只在用户需要时给出，保持少量、尖锐、可执行。

## Quality Bar

完成前必须检查：

1. 每个重话后面都有代码证据。
2. 没有把“能跑”偷换成“合理”。
3. 没有用“历史原因”给结构问题免责。
4. 没有停留在命名、格式、注释这些低价值层面。
5. 如果批判文档，至少击穿 2 个以上文档主张。
6. 如果批判代码，至少定位到一个真实的职责/边界/生命周期问题。

## Tone

1. 可以尖锐，但不能虚张声势。
2. 可以不留情面，但不能脱离证据。
3. 不要用“整体不错”“方向是好的”这类缓冲句，除非证据真的支持。
4. 允许直接说：
   - 这不是……而是……
   - 这段话在替……辩护
   - 这类结构靠约定活着
   - 这不是经验总结，而是经验粉饰

## Example Prompts

1. 请用最不留情面的方式锐评 BattleData，但必须结合真实调用链和写回路径。
2. 请结合当前仓库代码，对这篇 Notion 文档做一次逆向批判，指出它在哪些地方替旧设计辩护。
3. 请先严厉批判这个模块，再把最重的 3 个问题压缩成可执行改造方向。

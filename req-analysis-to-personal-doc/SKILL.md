# 需求分析到个人文档

## 目标
将“需求原文 + 代码现状”收敛为一份可执行的需求分析文档，并落地到个人目录，且可被团队通过相对路径直接访问。

## 范本
- 标准范本: [2026-04-24_政令系统_需求分析.md](../../../doc/6_个人目录/刘鸿森/01_需求分析/2026-04-24_政令系统_需求分析.md)
- 新分析稿在结构、粒度、链接写法、i18n Key 清单上需向该范本对齐。

## 适用场景
- 收到新需求文档（docx/md/xmind截图）后，需要快速形成研发可执行分析稿。
- 需要说明需求与现有Unity代码的影响范围、改动点、风险与验收口径。
- 需要把结果沉淀到个人目录并更新功能索引。

## 输入参数
- 需求来源路径（必填）
- 功能名（必填）
- 个人目录根路径（可选，默认: doc/6_个人目录/刘鸿森）
- 输出类型（可选）: 快速版 / 完整版

## 执行流程
1. 读取需求原文
- 优先读取原文件正文（docx 需先提取文本）。
- 保留原需求中的字段名、文案key、规则阈值。

2. 结构化拆解
- 提取: 目标、范围、核心规则、数据配置、UI需求、交互约束。
- 给出决策点: 触发条件、优先级、扣减/回滚时机、异常分支。

3. 代码结合分析
- 在仓库中检索相关模块、配置表、UI入口、协议或缓存层。
- 输出“现状 -> 需求变化 -> 影响文件/模块”。
- 若无法定位实现，明确“未发现对应模块”和建议落点。

4. 提取 i18n Key（必做）
- 扫描需求原文，识别所有形如 `xxx_yyy`、`xxx_fmt`、`xxx_tips`、`xxx_desc`、`xxx_label`、`xxx_btn` 等命名风格的多语言 key，以及任何被引号/反引号标记的文案占位符。
- 对每个 key 记录:
  - Key 名称（保持与需求原文一致，不擅自重命名）。
  - 用途说明（这条文案要表达什么）。
  - 使用位置（哪个界面/弹窗/气泡/按钮/列表项；如果需求里关联了具体模块或代码入口，一并标注）。
  - 占位参数（如 `{0} {1}` 的语义，例如 `{0}=月数, {1}=恢复值`）。
  - 显隐/触发条件（什么状态下展示）。
- 范本: 参见上方“范本”链接的“i18n Key 清单”章节。
- 若需求未给出 key 名，但描述了文案，需要在“待确认项”里列出“需要文案 key”的占位条目，不要凭空捏造 key。

5. 生成分析文档
- 在个人目录创建新文档，建议命名:
  - YYYY-MM-DD_功能名_需求分析.md
- 使用模板: [需求分析模板](./assets/需求分析文档模板.md)
- 文档必须包含可验证条目、待确认项、以及独立的 i18n Key 清单表。

6. 更新功能索引
- 在个人目录功能索引中追加或更新一行:
  - 功能名称
  - 需求文档链接
  - 分析文档链接
  - 代码入口
  - 状态/更新时间

6. 质量检查
- 检查链接均为仓库相对路径。
- 检查规则是否可测试（至少列出关键验收点）。
- 检查是否区分“明确结论”与“待确认项”。

## 分支逻辑
- 需求信息不足: 先给最小可行分析稿，并列“必须补充信息清单”。
- 代码线索不足: 先输出候选模块与检索依据，不臆造实现细节。
- 用户要求“先不动手”: 只输出分析，不创建文件。

## 完成标准
- 产出一份新分析文档（落到个人目录）。
- 功能索引已可点击跳转到需求文档与分析文档。
- 给出至少 5 条可执行验收点。
- 列出风险与待确认项。

## 推荐输出结构
1. 背景与目标
2. 范围（In/Out）
3. 规则与状态机
4. 数据与配置影响
5. 代码影响清单（文件/模块）
6. UI与交互影响
7. 风险与待确认项
8. 验收用例
9. 实施建议（阶段划分）
